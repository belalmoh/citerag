🧩 Component Importance Map

Each component rated by pipeline criticality — what breaks if it fails, and how central it is to the RAG quality.

🗂️
Critical — Tier 0

Indexer

The single point of truth between your documents and retrieval. Owns Qdrant collection lifecycle (create, upsert, delete). Every chunk lives or dies here. If the Indexer is down, zero queries can be answered — retrieval has nothing to search.

Criticality
Depends on: AsyncQdrantClientChunkembeddings
🧮
Critical — Tier 0

Embedder

Converts raw text into the mathematical representation that makes semantic search possible. Without embeddings, there is no "R" in RAG. Uses text-embedding-3-large (3072-dim) with manual batching and exponential-backoff retry. Quality of embeddings = quality of retrieval.

Criticality
Depends on: OpenAI APIAsyncOpenAI
🔍
Critical — Tier 0

Retriever

The bridge between a user's question and relevant knowledge. Queries Qdrant with HNSW approximate nearest-neighbor search (ef=128). Returns ranked SearchResult objects with scores, content, and source metadata. Garbage retrieval = garbage answers, no matter how good your LLM is.

Criticality
Depends on: IndexerQdrantquery_vector
✂️
High — Tier 1

Chunker

Structure-aware splitting powered by LangChain's RecursiveCharacterTextSplitter. Protects tables, code blocks, and lists from being split mid-structure using placeholder markers. Chunk size & overlap directly control retrieval precision vs. context completeness. Default: 1000-char chunks with 200-char overlap.

Criticality
Depends on: LangChainparsed text
📄
High — Tier 1

DocumentParser

The entry point for all knowledge. Uses pymupdf to handle PDF, DOCX, XLSX, PPTX, EPUB, and more. Extracts text, tables, and page-level metadata. Async-aware — offloads blocking I/O to a thread pool. If parsing fails, nothing downstream matters.

Criticality
Depends on: pymupdffile I/O
🤖
High — Tier 1

LLMClient

Generates the final answer from retrieved context. Uses GPT-4o-mini with exponential-backoff retry (3 attempts). Enforces a strict system prompt: "answer ONLY from provided context, don't make up facts." This is where hallucination is prevented.

Criticality
Depends on: OpenAI APIPromptBuilder
📝
High — Tier 1

PromptBuilder

Constructs the citation-enforced prompt. Wraps each chunk with numbered citations [1], [2] and appends source filenames. The system prompt demands "Cite your sources by referring to chunk numbers." This is the citation integrity layer — without it, answers lose traceability.

Criticality
Depends on: SearchResult[]
⚡
High — Tier 1

IngestionPipeline (Worker)

The orchestrator that sequences parse → chunk → embed → index. Runs as a Celery task with 3 retries. Bridges Celery's sync world to the async pipeline via asyncio.run(). Without this, documents never make it to the vector store.

Criticality
Depends on: CeleryParserChunkerEmbedderIndexer
📡
High — Tier 1

Query Endpoint (query.py)

The runtime orchestrator for every user question. Sequences embed → retrieve → build prompt → generate → log. Measures end-to-end latency. Returns structured QueryResponse with sources, token counts, and model info.

Criticality
Depends on: EmbedderRetrieverPromptBuilderLLMClientPostgreSQL
⚙️
Medium — Tier 2

Settings (config.py)

Central configuration via pydantic-settings. Loads from .env. Controls DB URLs, Qdrant connection, model selection, chunk sizes, and feature flags like enable_reranker. Every component reads from this.

Criticality
Depends on: .env file
🔗
Medium — Tier 2

Dependency Injection (dependencies.py)

FastAPI DI wiring. Caches Embedder and AsyncQdrantClient as singletons via @lru_cache. Creates fresh Indexer per request. Ensures resources are shared efficiently across the async event loop.

Criticality
Depends on: SettingsFastAPI
💬
Medium — Tier 2

Feedback API (feedback.py)

Collects 👍/👎 ratings per query with optional comments and relevant chunk IDs. This is the continuous improvement loop — feedback data feeds evaluation and future retrieval tuning.

Criticality
Depends on: QueryLogsPostgreSQL
📋
Support — Tier 3

Jobs API (jobs.py)

Polls document processing status (PENDING → PROCESSING → INDEXED → FAILED). Lets the frontend show upload progress. Observability, not core logic.

Criticality
Depends on: PostgreSQLDocument model
🔀
Planned — Not Yet Built

Hybrid Search (hybrid_search.py)

Will combine dense (vector) + sparse (BM25) retrieval for better keyword matching. Critical for exact phrases, names, and IDs that semantic search misses.

Depends on: Qdrant sparse vectorsBM25
🎯
Planned — Not Yet Built

Re-ranker (reranker.py)

Will apply a cross-encoder model to re-score top-K candidates. Typically improves relevance by 10–15% over pure vector similarity.

Depends on: cross-encoder modelRetriever
📊
Planned — Not Yet Built

RAGAS Evaluation (ragas_eval.py)

Will measure faithfulness, answer relevancy, context precision, and context recall. Target: faithfulness > 0.80, relevancy > 0.80.

Depends on: RAGAS librarytest queries

🏗️ Infrastructure & Storage

Three databases, each with a distinct role. Understanding why each exists is key to understanding the architecture.

🐘
Critical

PostgreSQL

Source of truth for everything relational. Stores document registry (filename, status, filepath), query logs (question, answer, tokens, latency), and user feedback (ratings, comments, relevant chunks).

Why not just Qdrant? Qdrant is optimized for vector search, not relational queries, joins, or audit trails. PostgreSQL handles the "business logic" data while Qdrant handles the "semantic" data.

Tables: documentsquery_logsfeedbackchunks
🔮
Critical

Qdrant

Vector database — the heart of semantic search. Stores 3072-dimensional embeddings with payload metadata (content, document_id, filename, chunk_index). Uses HNSW approximate nearest neighbor for sub-10ms retrieval.

Why Qdrant over Pinecone/Weaviate? Self-hostable, no vendor lock-in, built-in hybrid search support, excellent async Python SDK, and metadata filtering that lets you scope searches to specific documents.

Features: HNSWCosine distancePayload filtering
⚡
High

Redis

Celery message broker + future cache layer. Carries task messages between FastAPI and Celery workers. Will also serve as a query cache to avoid redundant LLM calls for identical questions.

Why Redis? Battle-tested as a Celery broker, in-memory speed, and dual-purpose: message queue + application cache in one deployment.

Roles: Celery brokerQuery cacheRate limiting
☁️
High

OpenAI API

External AI provider for both embeddings and generation. Embedding model: text-embedding-3-large (3072-dim). Chat model: gpt-4o-mini. Both have retry logic with exponential backoff.

Why OpenAI? Best quality/cost ratio for embeddings, excellent instruction-following for cited answers, and 128K context window handles large document contexts.

Models: text-embedding-3-largegpt-4o-mini

Indexer, Embedder, Retriever. If any of these fail, the system cannot answer queries. These are the non-negotiable core of RAG.

🟠 Tier 1 — Quality Path

Parser, Chunker, LLMClient, PromptBuilder, Pipeline, Query API. System still runs without them, but answer quality degrades severely.

Settings, DI, Feedback. Configuration, wiring, and improvement loops. Important for operations but not for real-time query serving.

Jobs API, planned components. Nice-to-have for monitoring and future enhancements. Not needed for core functionality.

💡 Architectural Insights

1️⃣

The Embedder is Used Twice

The same Embedder creates vectors during ingestion AND during query time. This is critical: if you used different embedding models for documents vs. queries, the vector spaces wouldn't align and retrieval would return garbage. The dependencies.py singleton ensures consistency.

2️⃣

Indexer is the Shared Backbone

The Indexer is the only component used by BOTH pipelines. Ingestion writes through it; retrieval reads through it. It's the single shared dependency between the two halves of the system. This is why it's Tier 0.

3️⃣

Citations Happen in the Prompt, Not the Model

CiteRAG doesn't rely on the LLM to magically cite sources. The PromptBuilder explicitly numbers every chunk [1], [2], ... and the system prompt demands citations. This is a design choice: structural guarantees over model hope.

4️⃣

Feedback Closes the Loop

Every query is logged with full traceability (tokens, latency, model). User feedback (👍/👎 + relevant chunk IDs) is stored and linked to query logs. This data is the foundation for future retrieval tuning and evaluation.

5️⃣

Celery Bridges Sync/Async Worlds

FastAPI is fully async, but Celery workers are synchronous. The IngestionPipeline.run() is async, and the Celery task wraps it with asyncio.run(). This pattern keeps the event loop free during uploads while still using Celery's battle-tested retry/monitoring.

6️⃣

Three Empty Files = The Roadmap

hybrid_search.py, reranker.py, and query_expander.py are stubbed out but empty. These represent the retrieval quality roadmap: hybrid (dense+sparse) → rerank → expand. Each one incrementally improves recall and precision.
